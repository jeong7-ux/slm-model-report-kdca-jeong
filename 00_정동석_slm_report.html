01모델 선택 이유

OpenAI Privacy Filter는 개인식별정보(PII) 탐지 및 마스킹에 특화된 토큰 분류 모델로, 공공기관 행정 업무에서 가장 민감한 과제인 개인정보 보호 문제를 직접적으로 해결할 수 있는 모델이다. 첫째, 공공기관은 민원 서류·내부 문서·정책 보고서 등 대량의 텍스트 데이터를 처리하며 이 과정에서 개인정보 유출 위험이 상존한다. 본 모델은 이름·이메일·전화번호·주소·계좌번호 등 8개 유형의 PII를 단일 포워드 패스로 탐지하여 높은 처리량(throughput)을 보장한다. 둘째, Apache 2.0 라이선스로 상업적·공공 목적 모두 자유롭게 활용 가능하며, 1.5B 전체 파라미터 중 50M만 활성화되는 Sparse MoE 구조 덕분에 온프레미스 환경이나 노트북에서도 실행 가능하다. 셋째, 128,000 토큰의 긴 컨텍스트 윈도우를 제공하여 장문의 행정 문서도 청크 분할 없이 일괄 처리할 수 있어 실무 워크플로우 효율화에 매우 적합하다.

03학습 데이터와 튜닝 방식

사전학습 단계: gpt-oss 아키텍처 기반의 자기회귀(autoregressive) 방식으로 사전학습된 체크포인트를 출발점으로 사용한다. 구체적인 사전학습 데이터 구성은 공개 범위 내에서 확인이 어려우나, gpt-oss 모델 카드를 참조할 수 있다. 주 언어는 영어이며 일부 다국어 강건성 평가가 보고되어 있다.

후속 학습(Post-training) 단계: 자기회귀 체크포인트를 양방향(bidirectional) 토큰 분류기로 변환하는 구조적 전환이 이루어졌다. 기존의 언어모델 출력 헤드를 제거하고 개인정보 레이블 분류 헤드(33개 클래스)로 교체한 뒤, 지도 학습 기반 토큰 수준 분류 손실(supervised classification loss)로 후속 학습을 수행했다. 이는 전형적인 SFT(Supervised Fine-Tuning) 방식에 해당한다.

핵심 차별점: 일반적인 MLM(Masked Language Model) 사전학습이 아닌, 자기회귀 모델의 사후 변환(post-training conversion) 방식을 채택하여 자기회귀 학습의 강점을 보존하면서도 양방향 분류 성능을 확보했다. 밴드 크기 128의 양방향 밴디드 어텐션(banded attention)을 적용하여 유효 어텐션 윈도우는 257 토큰이다.

04파라미터 및 구조적 특징

ArchitectureOpenAIPrivacyFilterForTokenClassification — gpt-oss 기반 Pre-norm Transformer Encoder-style 스택 + Sparse MoE FFN (128 experts, top-4)

05모델 파일 구성

핵심 모델 파일:
• config.json — 모델 아키텍처 설정 (레이어 수, 어텐션 헤드, MoE 전문가 수, 레이블 매핑 등 전체 하이퍼파라미터 정의)
• model.safetensors — BF16 형식의 모델 가중치 (약 1.4B 파라미터, safetensors 포맷으로 안전한 로딩 보장)
• model.sig — 모델 무결성 검증을 위한 서명 파일
• tokenizer.json / tokenizer_config.json — BPE 토크나이저 정의 (vocab 200,064, EOS 토큰 설정)
• viterbi_calibration.json — 제약 비터비 디코더의 전이 바이어스 파라미터 (정밀도/재현율 트레이드오프 조정용)

ONNX 변환 파일 (onnx/ 디렉터리):
• model.onnx + data 파일 — FP32 ONNX 추론용
• model_fp16.onnx — FP16 양자화 ONNX 버전
• model_q4.onnx / model_q4f16.onnx — INT4 양자화 (WebGPU 브라우저 실행용)
• model_quantized.onnx — 범용 양자화 버전

원본 체크포인트 (original/ 디렉터리):
• original/config.json / original/model.safetensors — 변환 전 원본 가중치
• original/dtypes.json / original/viterbi_calibration.json — 원본 데이터 타입 및 캘리브레이션 설정

06활용 가능 업무

1. 공공기관 문서 비식별화 자동화
민원 서류, 내부 보고서, 정책 자료 등에 포함된 개인정보(이름, 전화번호, 주소, 이메일, 계좌번호)를 자동 탐지·마스킹하여 개인정보보호법 준수 및 공공데이터 개방 시 비식별 처리 워크플로우를 자동화할 수 있다.

2. 대량 행정 데이터 정제 파이프라인
128K 토큰 컨텍스트 윈도우를 활용해 장문의 법률 문서, 감사 보고서, 회의록 등을 청크 분할 없이 일괄 처리 가능하다. 단일 포워드 패스로 전체 문서의 PII를 탐지하므로 배치 처리 효율이 매우 높다.

3. RAG(검색증강생성) 시스템의 전처리 필터
문서 인덱싱 전 개인정보를 사전 제거하여 RAG 기반 질의응답 시스템에서 개인정보 노출 위험을 차단하는 전처리 게이트로 활용할 수 있다.

4. 내부 챗봇 / 민원 응대 시스템 보안 계층
사용자 입력 텍스트에 포함된 개인정보를 실시간 탐지하여 로그에 기록되기 전 마스킹 처리함으로써 데이터 유출 사고를 사전에 방지할 수 있다.

5. 웹 브라우저 기반 경량 배포
Transformers.js + WebGPU(INT4 양자화)를 통해 별도 서버 인프라 없이 브라우저에서 직접 실행 가능하므로, 보안이 민감한 환경에서 데이터를 외부로 전송하지 않고 온디바이스 PII 탐지가 가능하다.

07한계와 주의사항

1. 한국어 지원 한계: 주 학습 언어가 영어이며 비영어·비라틴 문자 텍스트에서 성능 저하가 예상된다. 한국어 인명, 주소, 전화번호 형식에 대한 탐지 정확도는 파인튜닝 없이는 보장하기 어렵다.

2. 정적 레이블 정책: 8개 고정 PII 카테고리만 지원하며, 런타임에 레이블 정책을 동적으로 변경할 수 없다. 기관별 특수 개인정보 유형(예: 주민등록번호, 여권번호 등)을 탐지하려면 추가 파인튜닝이 필수적이다.

3. 과탐지/미탐지 위험: 흔하지 않은 인명 패턴, 지역별 명명 규칙, 이니셜, 경칭이 많은 참조에서 미탐지 가능. 공공기관명, 지명 등 공적 정보가 과잉 마스킹될 수 있다. 특히 혼합 형식 텍스트, 장문 문서, 구두점 과다 텍스트에서 스팬 경계가 분절되거나 이동할 수 있다.

4. 익명화 보증 불가: 모델 카드에서 명시적으로 이 모델은 '비식별화 보조 도구'이지 '완전한 익명화 보증'이 아님을 경고한다. 의료·법률·금융·인사·교육·정부 등 고감도 분야에서는 반드시 인간 검토(human-in-the-loop)를 병행해야 한다.

5. 인구통계적 편향: 학습 데이터의 지배적 분포와 다른 인구통계적·지역적·도메인 특성을 가진 텍스트에서 탐지 일관성이 떨어질 수 있으며 이는 공정성(fairness) 문제로 이어질 수 있다.

08종합 의견

OpenAI Privacy Filter는 SLM 범주에서 매우 독특한 포지션을 차지하는 모델이다. 일반적인 텍스트 생성 SLM과 달리 개인정보 탐지라는 특정 도메인에 최적화된 토큰 분류 모델로서, 공공기관의 개인정보보호 의무를 기술적으로 지원할 수 있는 실용적 가치가 높다.

도입 적합 조건: (1) 영어 기반 문서 처리가 주된 업무이거나, (2) 한국어 PII 데이터셋으로 파인튜닝할 역량과 자원이 있는 조직, (3) 온프레미스 배포가 필수인 보안 환경, (4) 대량 문서 배치 처리가 필요한 데이터 정제 파이프라인.

필수 POC 항목:
① 한국어 PII 데이터셋(주민등록번호, 한국식 주소, 한국 인명 등) 구축 및 파인튜닝 성능 검증
② 기관 내 실제 문서 샘플 1,000건 이상을 대상으로 정밀도(Precision)·재현율(Recall) 측정
③ 비터비 디코더 캘리브레이션 파라미터 조정을 통한 기관별 최적 운영점(operating point) 탐색
④ 기존 문서관리시스템(DMS)·전자결재 시스템과의 API 연동 검증
⑤ 처리 속도(throughput) 벤치마크: 온프레미스 GPU vs. CPU vs. WebGPU 브라우저 환경 비교

종합적으로, 본 모델은 Apache 2.0 라이선스, 경량 MoE 아키텍처, 128K 컨텍스트의 삼중 강점을 갖추고 있어, 한국어 파인튜닝을 전제로 공공기관의 개인정보 보호 자동화 인프라 구축에 핵심 컴포넌트로 채택할 가치가 충분하다.
